# Turn Craft — UI/UX 审计报告

> 审计日期: 2026-07-10  
> 审计范围: `/front/` 全部页面、CSS、交互逻辑、文案  
> 技术栈: Dioxus 0.7 (WASM) + 自写 CSS + Tailwind CSS v4.1
> 
> **视角前提**: 项目定位是"可无限扩展游戏类型 + 多人/AI 自由组队 + 用户自创游戏"的平台，本报告同时从终端用户观感和平台扩展性两个维度进行审计。

## 一、视觉体验

### 1.1 颜色搭配

**整体评价**: 暗色主题采用了统一的 slate+emerald 色系，CSS 变量体系设计良好，但存在多处硬编码颜色破坏一致性。

| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------- | ------ |

## 五、总结

### 按严重度分类统计

| 严重度 | 数量 | 主要问题 |
| ------ | ---- | -------- |
| **高** | 6   | 键盘快捷键反直觉、无确认删除、战绩数据伪造、空位检测逻辑不一致(两套)、德扑无响应式、public 页面不调 join API |
| **中** | 11   | 德扑配色脱离体系、内联样式过多、Tailwind 使用率低、CSS 命名混乱、红色气泡显示成功消息等 |
| **低** | 15   | 字体大小不一致、硬编码颜色、Backend URL 硬编码、注释代码残留、按钮样式缺失等及 CSS 代码质量问题 |
| **新增** | 7   | 无 Design Token 体系、无共享游戏 UI 组件库、slot/actor 命名无规范、`game_config` 无 schema、全量状态无增量推送、CSS 隔离无约定、`on_action` 无即时反馈 |

### 核心结论

1. **视觉层面**: 暗色主题的设计基础扎实，但德州扑克页面完全独立于全局设计体系——这不是单个页面的问题，而是**缺少 Design Token 体系**导致每加一个新游戏都可能重蹈覆辙。Emoji 图标降低了质感但在 Dioxus WASM 环境下是最轻量的图标方案。

2. **交互层面**: Enter 发送/ Ctrl+Enter 换行与主流习惯相反。缺少删除确认是数据安全风险。`on_action` 的 fire-and-forget 设计对简单操作足够，但当平台面向用户自创游戏时——用户写的游戏逻辑出错——前端无法即时感知失败。

3. **内容层面**: 个人主页的模拟战绩是最严重的内容准确性问题。注册成功用红色气泡显示是文案逻辑 bug。

4. **一致性层面**: 整体设计语言在应用壳层统一，但游戏层缺乏规范。CSS 代码质量（重复定义、命名混乱）目前通过复制粘贴传播——每加一个新游戏，技术债务会指数级增长。

5. **扩展性层面（新增）**: `GameEngine` trait + `GameUIRegistry` + `RoomActor` 的架构骨架是优秀的。但**缺少连接引擎和 UI 的中间层**——没有 Design Token、没有共享组件库、没有 slot 命名规范——导致新游戏的 UI 开发完全靠"读现有代码然后猜"。在 3 个游戏时这还勉强可控，到 10 个游戏时将是灾难。

## 六、可扩展性审计 — 面向"无限游戏类型 + 用户自创"平台的 UI 架构评估

### 6.1 现有插件架构盘点

项目拥有一套设计良好的游戏注册/分发机制，前后端各自独立但通过 `game_type` 字符串对齐：

```
┌─ 后端 ──────────────────────────────────────────────────────┐
│  GameRegistry<dyn GameFactory>                               │
│  ├── LincolnFactory   → LincolnEngine (platform_core)        │
│  ├── TexasHoldemFactory → TexasHoldemEngine                  │
│  └── WerewolfFactory  → WerewolfEngine                      │
│                                                              │
│  RoomActor 循环 (game_type 无关):                            │
│    engine.step() → Vec<EngineEvent> → broadcast to peers     │
│    EngineEvent::TriggerAi → AiWorker → LLM → step() again   │
│    EngineEvent::GameOver → Persist → Supervisor cleanup      │
├──────────────────────────────────────────────────────────────┤
│  platform_core::GameEngine trait (统一的引擎契约):            │
│    step(actor_id, action) → Vec<EngineEvent>                 │
│    to_json() / to_json_for_player(actor_id) → Value          │
│    to_ai_prompt(actor_id) → String                           │
│    current_actor() / is_finished()                           │
├──────────────────────────────────────────────────────────────┤
│  Actor/Slot 模型 (多人+AI 自由组合的基础):                     │
│    slot_configs: HashMap<"slot_name", "human"|"ai">          │
│    my_slot: 创建者占用的席位                                  │
│    human 席位 → 空 (等待真人加入) 或已占用                     │
│    ai 席位 → 创建时立即实例化 AI actor                        │
│    spectator → 不占席位但接收广播                              │
└──────────────────────────────────────────────────────────────┘

┌─ 前端 ──────────────────────────────────────────────────────┐
│  GameUIRegistry<GameUIDefinition>                            │
│  ├── "lincoln"      → icon + name + LobbyCard + GameComponent│
│  ├── "texas_holdem" → ...                                    │
│  └── "werewolf"     → ...                                    │
│                                                              │
│  GamePluginManager: game_type 分发 → 动态渲染 GameComponent   │
│  DynamicLobbyCard:  game_type 分发 → 动态渲染 LobbyCard       │
│                                                              │
│  GamePluginProps: 游戏组件契约                                │
│    state: Signal<Value>      — 服务端推送的 opaque JSON       │
│    on_action: Callback<Value> — 动作发送回调                   │
│    actor_id: String          — 当前客户端身份                  │
│                                                              │
│  WsBridge (通过 use_context):                                 │
│    opaque_state: Signal<Value>           — 全量状态信号        │
│    streaming_text: Signal<HashMap<..>>   — AI 流式输出         │
│    connected: Signal<bool>               — 连接状态            │
│    send(Value)                           — 动作发送            │
└──────────────────────────────────────────────────────────────┘
```

**这套架构的核心优势**：RoomActor 循环和 WebSocket 广播层完全与游戏类型解耦。新增游戏只需：
1. 在 `platform_core` 实现 `GameEngine` trait
2. 在 `backend` 注册 `GameFactory`
3. 在 `front` 注册 `GameUIDefinition`（LobbyCard + GameComponent）
4. 写 CSS

但第 4 步——**写 CSS**——是当前最脆弱的环节。

### 6.2 可扩展性的致命缺陷

#### 缺陷 A：无 Design Token 体系支撑新游戏 UI

目前 CSS 变量 (`--accent`, `--bg-deep`, `--text-primary` 等) 只覆盖了**应用壳层**的颜色。游戏内部组件（玩家座位、卡牌、动作面板、历史流水）没有任何预定义的 token。结果是：

| 游戏 | 颜色来源 | 与其他游戏的关系 |
|------|----------|-----------------|
| Lincoln | `game.css` — 继承了 `:root` 变量, 通过 `.bubble-content.judge/.pro/.con` 定义角色色 | 与 Werewolf 共享 `.bubble-row` 但角色色命名不一致 |
| Texas Hold'em | `poker.css` — **100% 硬编码**，不读取任何 CSS 变量 | 完全孤立 |
| Werewolf | 混合 `game.css` 的 bubble 样式 + 大量内联 `style: "background: var(--red)"` | 部分依赖全局变量但大量内联覆盖 |

**后果**：每新增一个游戏类型，开发者必须在"沿用 game.css 的类名"和"自写一套新样式"之间猜测。没有任何文档或约定告诉他们应该怎么做。

#### 缺陷 B：无共享游戏 UI 组件库

以下 UI 模式在 3 个游戏中都已出现，但**每次都是重写**：

| UI 模式 | Lincoln 实现 | Texas Hold'em 实现 | Werewolf 实现 | 应抽取为 |
|---------|-------------|-------------------|---------------|---------|
| 玩家名册/席位 | arena-sidebar 的 `.player-roster-card` | `.player-seat` (absolute 定位) | 内联 `players-status-bar` | `<PlayerRoster>` |
| 历史流水/时间轴 | `.bubble-row` + `.bubble-content` | `.history-panel` 内联样式 | 复用 `.bubble-row` | `<Timeline>` |
| 动作输入区域 | `.action-console` + `.console-textarea` | `.action-area` + `.action-buttons` | 复用 `.action-console` + 魔改 | `<ActionPanel>` |
| 回合/阶段指示器 | `.timeline-round` | `.phase-text` | `.timeline-round` + 内联 | `<PhaseIndicator>` |
| 空状态 | `.timeline-empty` | 无 | 无 | `<EmptyState>` |
| 流式输出光标 | `.streaming-bubble` + `.cursor-blink` (只在 Lincoln) | 无 (历史面板直接显示) | 复用 Lincoln 模式 | `<StreamingBubble>` |
| AI 思考内容切换 | `.toggle-ai-btn` 内联按钮 | `.toggle-ai-btn` 重复实现 | `.toggle-ai-btn` 再次重复 | `<AiContentToggle>` |

**当前做法**：一个新游戏开发者需要阅读 3 个现有游戏的几千行代码来推断"我应该怎么组织 UI"，然后从中复制粘贴。

#### 缺陷 C：slot/actor 命名无规范，全量状态无增量

| 问题 | 详情 | 影响 |
|------|------|------|
| 槽位命名**每游戏自定** | Lincoln: `Judge/Pro/Con`, Texas: `player1..6`, Werewolf: `Player1..7` | 前端 lobby card 必须硬编码 slot 名称枚举 |
| Actor ID 生成**工厂内自定** | `"human_pro"`, `"ai_player3"`, `"human_spectator_X"` | 前端判断 `kind == "ai"` 还要兼容大小写 (`Ai` vs `ai`) |
| 空席位表示**两处不同** | `lobby.rs` 用 `occupant.as_str() == "Empty"`, `public.rs` 用 `occupant.get("Empty")` | 已在 #31 中指出 — 新游戏复制哪处的逻辑？ |
| 全量状态同步 | 每次 `step()` 后广播整个 `to_json_for_player()` | 对手牌多/历史长的游戏 (德扑 6 人摊牌) 每次推送几 KB JSON |
| 无 action 结果反馈 | `on_action` 是 fire-and-forget，错误走 `alert()` | 游戏组件无法区分"发送成功等待中"和"服务端拒绝"，只能静默等下一次 state 更新 |

#### 缺陷 D：`game_config` 是裸 `Option<Value>`，无验证

德州扑克的 lobby card 在每个 `oninput` 事件中手动构造 `json!({"small_blind": ..., "big_blind": ..., "starting_chips": ...})`，且三个输入框各自独立触发——每次修改小盲注，都要重新序列化完整的 game_config JSON。新游戏的 lobby card 没有 schema 可参照，输入验证完全靠开发者自觉。

#### 缺陷 E：CSS 按游戏拆分但无隔离机制

当前 CSS 文件的组织：

```
front/assets/
├── main.css      # 全局：主题变量、侧边栏、toast、skeleton
├── tailwind.css  # Tailwind v4 (利用率 <5%)
├── login.css     # 登录页
├── lobby.css     # 大厅
├── game.css      # 游戏 arena 壳 + Lincoln + 流式输出
├── poker.css     # 德州扑克（完全独立色系）
├── settings.css  # AI 配置页
├── history.css   # 历史对局
├── profile.css   # 个人主页
├── public.css    # 公开房间
├── about.css     # 关于页面
└── replay.css    # 对局回放
```

问题：
- **`game.css` 不是纯"游戏壳"样式**，它混杂了 Lincoln 专属的 `.bubble-*` 类和 Werewolf 也在复用的 `.streaming-bubble`，分界线模糊
- **新游戏应该写自己的 CSS 文件还是扩展 `game.css`？** 没有约定。目前 poker 独立，lincoln/werewolf 共享 `game.css`
- **类名冲突风险**：`.player-seat` 在 `poker.css` 中，`.player-roster-card` 在 `game.css` 中——如果新游戏想复用 `.player-seat` 但需要不同样式，要么魔改 class，要么写内联覆盖

### 6.3 架构层面的正面评估

在批判之余，以下设计对"无限扩展"是**有利的**：

| 设计点 | 评价 |
|--------|------|
| `GameEngine` trait 的方法全用 `Value` 作为 I/O | 允许引擎内部使用任意类型的 state/action，不被序列化格式限制 |
| `to_json_for_player(actor_id)` 支持信息隐藏 | 德州扑克利用它隐藏对手手牌，狼人杀利用它隐藏身份——这是多人不对称信息游戏的基础 |
| `RoomActor` 循环与游戏类型完全解耦 | 新增游戏不用改一行 actor 代码 |
| `GameUIRegistry` 使用 `fn()` 指针而非 trait object | 零虚函数开销，编译期即可验证所有游戏都已注册 |
| `platform_core` 作为独立 crate | 类型定义在前端(replay 解析)和后端(引擎)之间共享，靠 serde 对齐 |
| `EngineEvent::PrivateMessage` | 已预留了"单播"机制，可扩展为角色专属通知（如狼人杀的夜间信息） |

### 6.4 面向"支持新游戏 + 用户自创"的建议优先级

按紧迫程度排序，聚焦于**不做这些之前无法有效扩展**：

| 优先级 | 行动 | 解决缺陷 | 预估工作量 |
|--------|------|---------|-----------|
| **P0** | 制定 Design Token 规范文档（游戏可用的 CSS 变量、色板、间距、字体 scale）并让 poker.css 迁移到 Token 体系 | A | 2-3 天 |
| **P0** | 抽出 `game-ui-kit` 组件：`<PlayerRoster>`, `<Timeline>`, `<ActionPanel>`, `<PhaseIndicator>`, `<StreamingBubble>`, `<AiContentToggle>` | B | 1-2 周 |
| **P1** | 统一 slot 命名规范和 actor ID 生成规则，写进 `platform_core` 文档 | C | 1 天 |
| **P1** | 为 `game_config` 定义类型化 schema（至少用 serde 的 `#[serde(deny_unknown_fields)]` 保护），让 lobby card 的输入验证有据可依 | D | 2-3 天 |
| **P1** | 增量状态推送（从全量 `to_json_for_player` 改为 `Vec<StatePatch>` 协议） | C | 1 周 |
| **P2** | CSS 隔离约定：每个游戏一个独立 CSS 文件，禁止跨游戏复用类名；定义共享工具类（`.game-btn-primary`, `.game-card` 等） | E | 2-3 天 |
| **P2** | Tailwind 决策：要么全面拥抱 Tailwind 并按需编译，要么完全移除，不要维持 5 个 utility 的鸡肋状态 | E | 1 天 |
| **P3** | 用户自创游戏 DSL/runtime 选型（JSON 配置 → WASM 沙箱 → Lua 脚本，三者成本差异巨大） | 新需求 | 架构调研 |
| **P3** | `on_action` 改为有返回值（`Result<Value, ActionError>`），让游戏组件即时感知动作是否成功 | C | 3-5 天 |

### 6.5 对原始报告中部分判断的修正

基于对架构的理解，以下判断需要调整：

| 原 # | 原结论 | 修正后判断 |
|------|--------|-----------|
| 5 (双栏侧边栏) | "空间浪费" | `arena-sidebar` 是游戏专用控制面板，不同游戏可能需要完全不同的侧边栏（如德州扑克不需要 AI 配置入口）。**不应移除，应让 arena-sidebar 也成为游戏插件的一部分**，让每个游戏定义自己的 `SidebarComponent` |
| 7 (max-width 不统一) | "缺乏节奏一致性" | 页面性质不同：大厅是 hub 页（需要展示多个信息流），settings 是表单页（窄宽度利于专注）。差异有合理性，但应明确定义规则（hub 页 1200、列表页 1000、表单页 800） |
| 18 (公开房间重复) | "功能重复" | 大厅右侧是快捷预览（无需离开创建流程就能看到活跃房间），PublicRooms 是完整浏览页。这是**分层信息架构**，不是重复。但空位检测逻辑必须统一 |
| 22 (大厅信息密度过高) | "新用户容易迷失" | 大厅本质是 hub 页。未来的优化方向不是减少内容，而是通过更好的视觉分层（progressive disclosure）降低认知负荷。新用户第一次进来可以只显示游戏选择 + 快速开始，高级配置默认折叠 |
| 25 (狼人杀"研发中") | "与事实不符" | 如果后端 waswolf engine 已完整，只是 UI 还没彻底完成，应改为 "Beta 测试中" 或直接移除标签 |
